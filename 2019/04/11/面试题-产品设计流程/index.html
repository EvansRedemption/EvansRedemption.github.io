面试题-产品设计流程

发表于 2019-04-11 | 分类于 交互面试题试答 | | 阅读次数:

假如您现在需要为一家企业做一款社交软件，请描述一个完整的产品规划到实施的过程包括哪些部分。

这是个看似很简单，但实际上又比较困难的问题，需要比较完备的知识框架，我尝试着答一下。
我发散开来答，不单纯的拘泥于产品规划。

先说一下产品设计的核心本质

  • 寻找问题
  • 分析问题
  • 解决问题

把这些核心展开，其实基本可以用经典软件研发模型SDLC来理解（System Development Life Circle）

  • 可行性
  • 分析
  • 设计
  • 实践
  • 测试
  • 维护

这是个衔尾蛇模型，最后一步维护又会发现新的问题，再回到可行性调研，周而复始。

继续把流程变成实际方法论

  • 用研调查
  • 分析问题
  • 概念设计
  • 原型设计
  • 项目评估

同样，项目评估后也又回到用研调查，无穷迭代。

首先，先问是不是，再问为什么。
这家企业需要做这个软件吗？
一个idea产生的时候，必然有原因，先去了解这个原因，确认它是不是拍脑袋想出来的。
比如说，我要和我的用户沟通、我要进军社交领域……所以我要做个社交软件。

有人来店里买个钻头，他想要个钻头吗？不，他是想要在墙上打个洞。他想要个洞吗？不，他是为了挂一幅画，那他是要挂一幅画吗？是的，他想要挂一幅画。那么，挂一幅画一定要打个洞吗？是不是可以用双面胶？是不是可以用其他方式？

这里有个小例子，“老年人在医院看病”的案例，院方想要做个APP来预约老年人看病、记录病例、管理治疗等等，看似确实满足老年人在看病时的需求，但没有考虑到老年人对于智能手机的使用情况。设计团队最后选择了一份经过设计的纸质文稿，辅以医疗流程优化来满足院方的需求。

因此我们要针对核心需求来寻找问题
对利益相关者制定的访谈，需要去探寻他们到底要什么？目的是什么？
对用户制定的访谈，需要去探寻他们的痛点、痒点是什么?

访谈也是个比较困难的点，往往直接问结论是的得不到正确答案的，因为他们也不知道。
需要结合自己的专业知识，从各个维度去串联一个场景，让用户/利益相关者描绘出一个笼统的内容。我们再从这些内容中抽丝剥茧，洞察核心。
真要我这样空口白话的描述如何做一份访谈，我做不到。（或者说，我自己真的去做一份用户访谈，也不一定能做好。）

Ok，做完相关访谈，如果你的访谈方式正确，你应该已经对设计目标做出了相应的修正。并且那些访谈结果都应该转化为了一个个设计机会点
以设计目标为主干，访谈内容为枝叶，向四周扩散出一个一个创新点，从而供养主干。
这里一定要注意，不能让枝干过于扩散，那样只会迷失你的方向。
此处，可以参照设计双钻模型。

此时我们就处在第四阶段，脑暴各种机会点的发散结果。

发散完了，我们手上应该会有一整套相关设计机会点及其解决方案。
其实这一部分就是我在开头提到的概念设计模块了，接下来你要做的就是去验证你的概念设计是否正确。可以通过快速原型测试或者A/B Test或者其他任何方式去和相关用户确认，你们的设计是否正确不能只停留在概念上。
但往往绝大多数公司，都忽略了这一步，或者说没有足够的资源去做这些事。

假设你通过测试验证了设计在大方向上没有问题，那此时就应该出具完整的交互原型稿、视觉稿进入开发流程。

开发上线以后，根据具体的数据、用户需求的变更、业务目标的变更等等因素，进行二次迭代、三次迭代…

后记

以上是比较学院派的设计流程。
我讲这么多，基本算是把设计双钻模型掰碎了来讲，有一定专业知识的人，其实看到设计双钻模型基本就知道大流程是怎么样的了。

当然也有其他设计流程，比如设计冲刺之类的，这一块我没接触太多，以后再补充吧。

再回过头来吐槽一下我现在的工作，基本就是项目组提出什么需求，我们用原型图画出来而已，一个纯粹至极的画图员。
调研？嗯，偶尔会有，但这种调研就像我上文说的，你只是在直接问对方想要什么。能调研出东西就真的见鬼了。
设计机会点发散？不存在的，时间和资源完全不够。
倒也不全都是坏处，它能锻炼我对于需求落地的能力，快速处理能力。我也一直认为落地是一个交互设计师最最基础的能力，只有在保证了落地以后，才能去考虑更多创新设计，否则连最基本的逻辑都搞不定，还妄想什么？
